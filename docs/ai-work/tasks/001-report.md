# Task 001 — report

## The result, in plain language

The CLI now tells the truth about whether a folder really is a Cairn project.

Before this task, `cairn status` treated *any* `AGENTS.md` that contained the word
"Cairn" anywhere as a real Cairn project — even a folder whose `AGENTS.md` just said
"We follow Cairn-style ideas." It would then print a hollow, made-up dashboard.

Now the CLI recognises a real contract by its *structure*, not by a stray word. A
folder only counts as a Cairn project when its `AGENTS.md` carries all three markers
that every genuine Cairn contract shares:

1. the `# Project Contract` heading,
2. a `Contract v<number>` version string (e.g. `Contract v1.2`), and
3. the project-facts labels `STATUS:`, `PROJECT NAME:`, and `CURRENT MILESTONE:`.

A casual mention of "Cairn" now correctly gets the "No Cairn contract in this folder…"
message and a non-zero (error) exit code, while this real repository — and any freshly
`cairn init`-scaffolded project — is still recognised and shows its true dashboard.

### One honest note about the marker I used

The brief suggested the version marker `Cairn Contract v` (e.g. `Cairn Contract v1.2`).
While building, I found that in the real `AGENTS.md` and `CONTRACT-TEMPLATE.md` the word
"Cairn" ends one line and "Contract v1.2" begins the *next* line (it wraps), so the
literal phrase `Cairn Contract v` never appears as one continuous piece of text. Had I
tested for that exact phrase, the fix would have wrongly rejected the real contract. The
brief explicitly left the exact combination to the builder ("The builder chooses the
exact combination, but it must satisfy all three anchor cases"), so I used the reliable
`Contract v<number>` marker instead — the same version pattern the existing
`parseFacts` function already relies on. One of the two new tests deliberately reproduces
the line-wrapped layout to guard against this exact mistake in the future.

## Files changed

- `cli/src/files.ts` — rewrote `isCairnProject(root)`. Same name and signature
  (`(root: string) => boolean`); the three commands that call it (`status`, `init`,
  `task`) are untouched and keep calling it the same way.
- `cli/test/files.test.ts` — added two tests:
  - "isCairnProject rejects an AGENTS.md that only mentions Cairn" (the bug case);
  - "isCairnProject accepts a real contract even when the version wraps a line"
    (the regression guard described above).
- Rebuilt output under `cli/dist/**` and `cli/assets/contract.md` was regenerated by the
  build step. These are git-ignored artefacts and are **not** part of the commit.

Nothing outside `cli/` was edited. No file was deleted, moved, or renamed. The
`status.ts` "No Cairn contract…" message was left exactly as-is (it was already correct).

## How the owner can see or try the result

You do not need to change anything; just look. (See "How do I try it?" if you'd like the
CLI to hand you these steps again.)

1. **See the true dashboard.** In this project's folder, run:

   ```
   node cli/dist/src/index.js status
   ```

   You should see "Cairn (Contract v1.2, ACTIVE)" and the milestone line. Success looks
   like real project facts; failure would look like "Unnamed project (Contract v?, ?)".

2. **See the bug is gone.** Make a new empty folder somewhere, create a file in it
   called `AGENTS.md` containing one line such as `We follow Cairn-style ideas.`, then
   run `cairn status` from inside that folder (or
   `node "<path to this repo>/cli/dist/src/index.js" status`). It should say
   "No Cairn contract in this folder…" and print no dashboard. Success looks like that
   plain message; failure would be a fake dashboard appearing.

## Limitations and remaining uncertainty

- The check proves the detection logic behaves correctly for the fake case, the real
  contract, and a fresh scaffold. It **cannot** prove that every possible unusual
  `AGENTS.md` in the world is classified perfectly. A contrived file that happened to
  include all three structural markers would still be treated as a contract — but that is
  a deliberate, contract-shaped file, not a casual mention, which is exactly the line this
  task set out to draw.
- Command output proves the named checks ran and passed. It does not prove the CLI is
  flawless in every scenario, only that these specific behaviours now hold.
