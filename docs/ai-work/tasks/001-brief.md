# Task 001 — brief

## Lane: Standard · Mode: Final

**Why Standard (not Tiny):** the fix changes one small function, but that function
(`isCairnProject`) is shared by three different CLI commands (`status`, `init`,
`task`). Making it stricter could accidentally stop recognising a *real* Cairn
project, so it needs new tests and careful checking. Uncertainty like that moves the
work up to the Standard lane. It is **not** High-Stakes: nothing here touches
dependencies, stored-data formats, security, payments, deployment, or any outside
service — it is a self-contained bug fix in the CLI, covered by unit tests.

**Why Final (not Draft):** there is a single correct behaviour to implement (a fake
contract should be recognised as fake). This is a straight bug fix meant to become
the CLI's real behaviour, not a candidate to park for the owner to choose between.

## The visible outcome

Today, if you run `cairn status` inside any folder that happens to have an `AGENTS.md`
file containing the word "Cairn" *anywhere* — even a stray mention in an unrelated
project — the CLI pretends a real Cairn project is there and prints a hollow
dashboard like:

```
Unnamed project  (Contract v?, ?)
Milestone: not set
```

After this task, `cairn status` in that same folder will instead say plainly that
there is no Cairn project here, for example:

```
No Cairn contract in this folder. Run `cairn init` (empty folder) or see PROJECT-CONVERSION.md.
```

...and it will exit with a non-zero (error) code, exactly as it already does for a
folder with no `AGENTS.md` at all.

Crucially, a **real** Cairn project must keep working unchanged: running
`cairn status` in this very repository must still show the true dashboard (project
name "Cairn", status ACTIVE, the current milestone, recent work), and a freshly
created project (via `cairn init`) must still be recognised as a Cairn project.

## How it moves the current milestone

The milestone is: *a real-model `cairn task` completes an improvement to Cairn
itself, end to end.* This is a concrete, real improvement to Cairn's own CLI, carried
through the full Define → Build → Verify → Decide loop. It also removes a rough edge
that would confuse the very beginners Cairn is built for. The builder's report will
give the honest YES / NO / UNCLEAR call on milestone movement.

## What "a real Cairn contract" means (the test the fix must apply)

An `AGENTS.md` counts as a real Cairn contract only if it carries the structural
markers that both the real contract and every freshly-scaffolded project share — not
just the word "Cairn." Reliable markers present in both `AGENTS.md` and
`CONTRACT-TEMPLATE.md` include:

- the heading `# Project Contract`,
- the version string `Cairn Contract v` (e.g. `Cairn Contract v1.2`), and
- the project-facts labels `STATUS:`, `PROJECT NAME:`, and `CURRENT MILESTONE:`.

The builder chooses the exact combination, but it must satisfy all three anchor cases
in the DONE checklist below. A single casual mention of "Cairn" must **not** be
enough.

## What may change

- `cli/src/files.ts` — rewrite the `isCairnProject(root)` function so it recognises a
  real contract by its structure, not by a bare "Cairn" mention. Keep the same
  signature: `(root: string) => boolean`.
- `cli/test/files.test.ts` — add tests: (a) a folder whose `AGENTS.md` only mentions
  "Cairn" is rejected (`isCairnProject === false`); (b) a real/ scaffolded contract is
  accepted (`isCairnProject === true`). Existing tests stay green.
- Rebuilt output under `cli/dist/**` may be regenerated by the build step (it is
  git-ignored and is not committed).

## What must NOT change

- The function's name and signature, so `status`, `init`, and `task` keep calling it
  the same way.
- The behaviour for a genuine contract: this repo's `AGENTS.md` and any
  `cairn init`-scaffolded project must still return `true`.
- The existing "No Cairn contract in this folder…" message in
  `cli/src/flows/status.ts` is already correct; it may stay exactly as-is. (A small
  wording polish is allowed but not required, and no other behaviour there changes.)
- Any file outside `cli/` — no edits to the contract, docs, the app (`cairn.html`),
  or anything in `docs/`. No files deleted, moved, or renamed.
- The git-ignored build artefacts (`cli/node_modules/`, `cli/dist/`,
  `cli/assets/contract.md`) are touched only as normal build output, never by hand.

2. **Real projects still work.** Run `cairn status` in this repository — it still shows

## What DONE requires

2. `cairn status` in this repository still prints the true dashboard (name "Cairn",

3. A freshly `cairn init`-scaffolded project is still recognised (the existing

4. `npm run build` succeeds and `npm test` passes, including the two new tests.

- **`BUILD_OR_TESTS_FAIL`** — the build or the test suite cannot be made green within
  this brief's boundary.

A STOPPED task gets no success commit; the state is preserved exactly and reported.
