# Cairn 0.0.1 Formal Reset — Design

Date: 2026-07-22
Status: approved by owner (this session)
Scope: documentation, legacy archive, and version identity only. No behavior change.

## Why

Cairn's history spans three governance generations (Contract v1.x gates, v2.x
risk-based autonomy and amendments, v3.0 serial reset). An audit of all 48 task
sessions (000–047) and every document found:

- **Paperwork was the main failure mode.** Every real Codex Exec run (036, 038,
  040, 041, 043, 047) stopped on record verification —
  `RECORD_VERIFICATION_FAILED` or `MODEL_RECORDS_MISSING` — never on the product
  change. No report states whether the requested smoke file was created;
  verification checked paperwork, not outcomes.
- **Total-discard privacy made failure undebuggable.** Five task cycles plus
  three diagnostic tasks yielded one unconfirmed hypothesis (the sandbox blocks
  file writes in non-interactive exec mode).
- **Ceremony was flat, not proportional.** One-line fixes carried 150–450-line
  briefs. Allowed-file whitelists stopped working features over trivia while a
  real security flaw passed every procedural check. The largest build (Cairn
  Desktop, +12,960 lines) bypassed the gated loop entirely.
- **Self-reported metrics bent when they bound.** The Direction Gate punished
  honest "milestone: NO" answers until reports rationalized YES.
- **The docs define Cairn by negation.** Seven top-level files enumerate removed
  machinery (Direction Gate, Experimental Draft, Bootstrap mode, schedulers,
  parallel lanes) that a new reader has never seen. Stale claims persist:
  GETTING-READY.md says the build stops before any real codex exec call (false
  since Task 034); README points at machine-local build artifacts.

**What consistently worked, and survives the reset:** honest DONE/STOPPED
discipline with retained evidence; the append-only one-row-per-task LOG;
exact-path commits; protection of the owner's starting Git state (no user work
was lost in 48 tasks); failing-test-first evidence; $0 mock-mode verification;
one plain confirmation at a concrete risk boundary; fresh-context review, which
caught real defects every time it ran (013, 016, 019, 028).

## Decisions (owner-approved)

1. **Contract 0.0.1 is a fresh rewrite.** The v3.0 text moves to legacy.
2. **Legacy means `docs/legacy/` plus a git tag.** Old docs and task records
   move there, organized and untouched. The pre-reset commit gets tag
   `legacy-v3.0`. Task numbering restarts at 001 with a fresh LOG.
3. **The reset touches docs and version strings only.** Product fixes informed
   by the audit become the first numbered tasks of the new era.

## Contract stances (the fresh rewrite)

The v0.0.1 contract stays roughly one page, written entirely in positive
language. Its positions, resolving the audit's contradictions:

1. **Records are memory; outcomes get verified.** Verification asks "does the
   requested visible outcome hold?" — never byte-matching of paperwork. (Moving
   record authorship from the model to Cairn is a future code task; the
   contract simply stops mandating the broken alternative.)
2. **Intent boundaries, not file whitelists.** Name what must not change
   (behavior, dependencies, data, security posture). Require disclosure of
   every file touched. Obviously correct adjacent fixes are allowed with
   disclosure.
3. **Reviews are optional and advisory**, recommended before anything large or
   irreversible.
4. **One serial loop; big work plans first.** Large changes get a design or
   plan document, then land as serial recorded tasks.
5. **Keep the proven core:** concrete-risk-boundary confirmations, Git
   protection, repair-in-task, and the append-only log — with the log table
   format defined canonically in the contract.
6. **Attempt limit.** Two stopped attempts at the same goal force a step-back
   diagnosis before any third attempt.
7. **Portability rule.** Canonical docs never cite machine-local or gitignored
   paths.

## Execution order

1. **Close the books.** Verify the Task 045 test passes against a fresh build,
   then commit Task 045 (CHANGELOG.md, README.md, app/tests/routing.spec.ts,
   app/vite.main.config.ts, the 045 records, and the 045 LOG row) and a second
   commit for the 046/047 records ("046 superseded by 047" in the message).
   Push the unpushed commits — this work has no remote backup.
2. **Tag `legacy-v3.0`** on the resulting HEAD. The name deliberately avoids the `v*`
   pattern: the release workflow triggers on `v*` tags and would fire the
   first-ever installer build. Tag `v0.0.1` only when a release is intended.
3. **Legacy move.** Build `docs/legacy/` as above and write its index.
4. **Contract rewrite.** New CONTRACT-TEMPLATE.md and AGENTS.md. Rebuild core
   so `core/assets/contract.md` and `app/resources/contract.md` regenerate.
   Update the cairn.html embed. Delete `cli/assets/contract.md` and its
   .gitignore line. Update the fixture literal in `core/test/serial.test.ts:39`.
   The parsing regexes in `core/src/files.ts` already accept "Contract v0.0.1";
   no code change.
5. **Doc rewrites** per the disposition table.
6. **Version bumps.** Set app, core, and cli package.json to `0.0.1`; run
   `npm install` to refresh the three lockfiles. Nothing pins the old versions
   and no GitHub releases exist, so the in-app update check is unaffected.
7. **Verify.** Core, cli, and app test suites; a mock-mode smoke run; a
   mirror-equality diff (template ↔ core asset ↔ cairn.html embed).
8. **Record the reset as Task 001** of the new era — the first row of the
   fresh LOG.

Optional cleanup, owner's call: delete superseded build output
`app/out/Cairn-win32-x64/` and `app/out/make/` (~484 MB); keep
`app/out/repaired/` (483 MB) only if the old 0.1.0 installer still matters.

## Out of scope: recommended first tasks of the 0.0.1 era

1. **Reproduce the Codex failure in the open.** Run `codex exec` with the same
   sandbox flags from a clean PowerShell (not nested in a Codex session) with
   visible output; confirm whether `workspace-write + on-request` blocks all
   file writes non-interactively. Diagnose first, then wire the proven
   invocation into the app.
2. **Cairn authors all task records**; the model only makes the product
   change. This eliminates the failure class that consumed tasks 040–047.
3. **Keep raw provider output in a gitignored local debug log** (credential-
   shaped tokens redacted) with a "view raw log" affordance on stopped
   results.
4. **Automated mirror-equality test** for the contract copies.
