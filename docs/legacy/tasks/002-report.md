# Task 002 — report

## The result, in plain language

The live status line during a `cairn task` can no longer wrap, so it can no longer
flood the console.

Before this task, Cairn fed the AI's streamed words into the spinner cut to a fixed 76
characters. With the spinner's own 3-character prefix that is about 79 columns. In any
console window narrower than that — like the one in the bug report, where the line was
clipped mid-word at "rewritte" — the line wrapped onto a second row. The spinner
redraws itself by jumping to the start of "the" line and erasing it, but it measures
lines by counting newline characters and never asks how wide the terminal is; once the
text wrapped, its erase landed on the wrong row and every animation frame (about 8–12 a
second) left a fresh copy marching down the screen. That is the wall of repeated
`The diff is exactly within the brief's boundary: \`isCairnProject\` rewritte…` lines.

Now every status line passes through one small helper, `spinnerLine`, that collapses
any newlines to spaces and cuts the text so the finished line — prefix included — stays
strictly inside the terminal's current width, with a column to spare. A line that
cannot reach the edge cannot wrap, so it cannot flood. The status line still animates
and still shows a readable, live summary of each step; it is just bounded now.

## Files changed

- `cli/src/ui.ts` — added one exported helper, `spinnerLine(raw, columns?)`. It
  collapses whitespace, defaults an unknown width to 80 columns, and cuts the text to
  `width − 4` characters (3 for the spinner prefix, 1 for margin). The existing exports
  in this file are unchanged.
- `cli/src/flows/task.ts` — the shared `events()` helper now routes **both** live
  callbacks, `onText` and `onTool`, through `spinnerLine(raw, process.stdout.columns)`.
  The old hardcoded `.slice(0, 76)` and `.slice(0, 60)` are gone. Reading
  `process.stdout.columns` on each call means the bound follows the window even if it is
  resized mid-run. Nothing else in the file — the loop order, the approval gate, the
  role policies — changed.
- `cli/test/ui.test.ts` — new test file with five tests covering the fix's promise (see
  below).
- `cli/package.json` — added `dist/test/ui.test.js` to the `test` script so the new
  tests run. This is a script-line change only; **no dependency was added, removed, or
  upgraded** (`@clack/prompts` stays at 0.11.0).
- Rebuilt output under `cli/dist/**` and `cli/assets/contract.md` was regenerated by the
  build step. These are git-ignored artefacts and are **not** part of the commit.

Nothing outside `cli/` was edited except this task's own `002-brief.md` / `002-report.md`.
No file was deleted, moved, or renamed. The uncommitted task-001 row in
`docs/ai-work/LOG.md` was left exactly as it was — the build never touched it.

## How the owner can see or try the result

You do not need to change anything to be safe; here are two ways to see it.

1. **Offline, no model call (fastest).** In `cli/`, run the test suite:

   ```
   npm test
   ```

   Look for the five `spinnerLine` tests passing, including
   "spinnerLine no longer floods a narrow console with the reviewer's line." Success
   looks like `pass 20 / fail 0`; failure would be any `✖` on a `spinnerLine` line.

2. **Live, in a narrow window.** Make your console window narrow (drag it to roughly 70
   columns wide), then run a real task and take it to the Verify step:

   ```
   node cli/dist/src/index.js task
   ```

   As the reviewer streams, watch the status line: it should stay a single row that
   updates in place. Success looks like one animated line; failure would be the old wall
   of repeated `The diff is exactly within the brief's boundary…` lines scrolling past.

## What still needs a human check

- The **live narrow-window run** (option 2 above) is the one check an automated test
  cannot fully stand in for, because the flood is a real terminal's wrapping behaviour.
  A quick look — narrow the window, run to Verify, confirm one line instead of a wall —
  is worth doing once to see the fix with your own eyes.

## Limitations and remaining uncertainty

- The tests prove the invariant that removes the flood: at widths 40, 72, 80, 120, and
  unknown, the rendered status line stays strictly inside the terminal width and carries
  no newline. They do **not** drive a real animated spinner inside a wrapping terminal —
  that is what the manual narrow-window run covers.
- The fix controls the text Cairn hands the spinner; it does not change the
  `@clack/prompts` library, which still measures lines by newline count. If a future
  change fed the spinner text some other way (bypassing `spinnerLine`), the same wrap
  could return. The single helper is the one place to keep this safe.
- At an absurdly tiny width (a handful of columns) the message is cut to almost nothing
  but still will not wrap; this is a pragmatic floor, not a case that occurs in normal
  use.
- Command output proves these specific checks ran and passed. It does not prove the CLI
  is flawless in every terminal, only that the wrap-and-flood behaviour is now prevented
  at every width tested.
